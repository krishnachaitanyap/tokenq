🏢 Proposed enterprise — additional hop to LLM gateway

Enterprises typically don't want raw Anthropic API keys distributed to laptops. They route all LLM traffic through a corporate gateway that handles SSO authentication, audit logging, DLP, rate limiting, and multi-vendor routing (Anthropic / Azure OpenAI / AWS Bedrock). The gateway holds the real provider keys.

Good news: the only required change is one env var — TOKENQ_UPSTREAM already exists (config.py:4). Point it at the gateway and tokenq forwards there instead of Anthropic. The 8-strategy pipeline still runs locally — compression, caching, and bandit decisions all happen before traffic leaves the laptop.

What stays vs what's added

🔵 unchanged from current
  • All 8 pipeline strategies run identically
  • Local SQLite at ~/.tokenq/tokenq.db
  • Per-developer dashboard on :8090
  • Cache hits never leave the laptop
  • Compression happens before egress (less WAN, less gateway load)
  • Zero changes to client integration code
🟢 new in enterprise mode
  • TOKENQ_UPSTREAM points at gateway URL
  • Header injection from local config (TOKENQ_FORWARD_HEADERS)
  • Optional mTLS client cert (TOKENQ_MTLS_CERT)
  • Auth header carries SSO token, not Anthropic key
  • Optional telemetry sink for centralized cost dashboards
  • (Gateway side: SSO, audit, DLP, multi-vendor routing)
Why tokenq belongs before the gateway, not behind it: compression, cache hits, and bandit decisions all need to happen before traffic crosses the corporate WAN. A cache hit on the laptop is a 0ms response; a cache hit at the gateway is still a network roundtrip. tokenq's compression also reduces WAN bandwidth and gateway load — the gateway sees an already-trimmed request body, so its rate limits stretch further.

🎯 Migration path

1. Rolling out to one team (week 1).
Pilot team installs tokenq the consumer way (tokenq install + ANTHROPIC_BASE_URL). Validate that the dashboard surfaces actual savings on their workload. No gateway needed yet.
2. Stand up the gateway in shadow mode (week 2-3).
Deploy the LLM gateway alongside the existing direct-to-Anthropic flow. Have it validate SSO tokens but pass through unchanged. Monitor for auth failures and request-shape compatibility issues.
3. Flip TOKENQ_UPSTREAM for pilot team (week 4).
Push a new env config to pilot team's laptops. They restart Claude Code; nothing else changes for them. Verify dashboard still works (tokenq's local DB is unaffected by upstream changes).
4. Enable gateway-side governance (week 5+).
Turn on DLP scanning, per-team rate limits, audit logging. The gateway can now reject specific request shapes — tokenq's pipeline still ran, so even rejected requests have local telemetry.
5. Centralized cost rollup (week 6+).
Optional: ship per-employee tokenq DBs to a central observability sink. The gateway already knows total spend; tokenq adds the *strategy-level breakdown* (which compression / which cache hits / which bandit decisions saved how much) per developer.

🎰 The casino analogy

Think of three slot machines. Each one charges a different amount per pull, and each one pays out at a different rate. The bandit's job: figure out which to play, without spending too much on the bad ones.

What you're seeing: on every pull the bandit picks the arm whose "guess" is highest right now (using Thompson sampling). After a few pulls you'll see it converge on the cheap-but-good arm — rarely picking Opus once it has enough evidence Haiku also works.

🎯 Thompson sampling — picking with appropriate confidence

Instead of always picking the model with the highest average success, Thompson sampling rolls a random number from each model's belief curve. Models we've barely tested have wide curves (so they sometimes get picked anyway — we need to learn about them). Models we've tested a lot have narrow, confident curves.

Click Sample to roll a random θ from each curve. Highest θ wins this round.

💰 The reward function — how a "win" is scored

After every call, tokenq scores the outcome 0.0 → 1.0. The score blends did it succeed? with did it cost less? — so an arm that "succeeds expensively" doesn't dominate one that "succeeds almost as often, much cheaper".

stop_reasonsuccess scoremeaning
end_turn1.0finished cleanly ✓
tool_use0.9called a tool — usually fine
max_tokens0.2got cut off — bad sign
error / hard fail0.0no credit for "failing cheaply"
anything else0.5ambiguous
Final reward = 0.7 × success + 0.3 × cost_score
where cost_score = 1 − min(cost/$0.05, 1) — capped so a single huge call can't punish an arm forever.

🛡️ Why this design is safe

1. Never upgrades.
Your chosen model is the ceiling. If you ask for Haiku, no routing. If you ask for Opus, the bandit can pick Haiku/Sonnet/Opus — never anything more expensive.
2. Shadow by default.
Day 1 of installing tokenq, every request gets a counterfactual recommendation logged — but the request itself is unchanged. You collect evidence with zero risk.
3. Graduation gate.
A cheap model needs ≥30 samples + ≥0.6 mean reward + ≥30% cost savings before it ever sees a live request.
4. Demotion.
If a graduated arm's rolling reward drops below 0.4, it's auto-demoted back to probation. (0.55 if it was promoted from shadow data — tighter floor for tokenq's own picks.)
5. 5% exploration.
Live mode occasionally picks an unproven arm so it can earn graduation. Without this, the system would freeze on whatever happened to graduate first.
6. Bucketed.
Each request "shape" learns separately — Haiku might graduate for short prompts but never for tool-use, and that's fine.
Source: src/tokenq/pipeline/bandit.py — 692 lines, fully tested in tests/test_pipeline_bandit.py. Tunable via env vars: TOKENQ_BANDIT_MODE, TOKENQ_BANDIT_GRADUATE_MIN_N, TOKENQ_BANDIT_EXPLORE_RATE, etc.

🧊 Why it pays out forever — the cache stability invariant

The trick: when we cut messages, we always snap to the same chunk boundary. So the resulting prefix is byte-identical across every turn until the next rollover. Anthropic's prompt cache hits → 0.1× rate on those bytes.

8 turns: 1 rollover (cache_creation) + 7 cache hits
The recurring win: we pay for one cache rebuild on the rollover turn, then reap (dropped_tokens − summary_tokens) in savings on every subsequent turn until the next rollover. That's why compaction_events stores per-turn savings, not per-request.

🛡️ The orphan tool_result guard

Edge case: if we cut at a position where the new first message starts with a tool_result whose paired tool_use we just dropped, the API returns messages.0.content.1: unexpected tool_use_id. So we walk forward past such messages. 

✗ INVALID: cut here →  | user(tool_result for X) | assistant(...) | user(...) |
                            └─ orphan, no matching tool_use prior

✓ VALID:   cut here →    | user(...)               | assistant(...) | user(...) |
                            └─ walked forward past the orphan

🎯 The "only credit last-message savings" rule

This is the subtle correctness bit. We mutate every tool_result every turn — that's what keeps the byte-stable prefix the cache needs. But only tool_results in the last message are new; the rest were already credited on previous turns and are now coming from the cache. Counting them again would over-count savings.

Turn N has 4 messages:

tool_result (NEW)
↑ ONLY this saves credit
Why we mutate them all anyway: if we only compressed the last message, the byte stream of earlier messages would diverge between turns (different fixed timestamps in ANSI sequences, etc.) — destroying the prompt cache. So we apply the same compression to every block, but only credit the delta from the last message in saved_by_compress.

The 5 Anthropic pricing axes

Anthropic prices per million tokens, not per call. A "token" is roughly 0.75 words of English (or ~4 characters of code). Rates shown are Opus base prices. Sonnet is 1/5, Haiku is 1/15. The cache multipliers apply on top of whichever model you're using.

Quick math: a typical Claude Code turn might send 50,000 input tokens (system + tools + history + cache reads) and generate 4,000 output tokens. At Opus rates that's 50,000 × $15/1M + 4,000 × $75/1M = $0.75 + $0.30 = $1.05 per turn. Multiply by ~150 turns/day → ~$157/day. That's why the per-million rates feel small but the bill isn't.

🧩 How they compound

1. Strategies stack along the cache axis.
Compactor + Compressor + Dedup + SkillLoader all leave smaller bytes behind. Every cache_read benefits from all of them at once, multiplied by the number of cache-hit turns. That's why the cache-read axis (the cheap one) is the largest savings opportunity in absolute dollars.
2. ExactMatchCache short-circuits everything.
A hit means no upstream call at all — saves on every axis simultaneously. But hits are rare on agentic workloads (most prompts differ in some byte). Best for templated/Q&A flows.
3. BanditRouter multiplies all of the above.
Routing Opus → Sonnet doesn't shrink any axis, but it changes which rate applies — so every input/output/cache token costs 1/5. Stack with the trim strategies for compounding wins.
4. OutputController is the only direct attack on output.
Every other strategy nibbles at input/cache axes. Output is 5× input, but it's a smaller fraction of the bill on cache-heavy agent workloads (~20% on this 7d sample). Highest-leverage on Q&A workloads where cache reuse is low.
5. Cache rebuilds are the only "watch out".
Compactor rollovers, SkillLoader changes to system, BigMemoryInject first-turn — each forces one Anthropic-side cache_creation at the 1.25× rate. The recurring 0.1× cache_read savings have to outweigh that one-time cost. The default thresholds are tuned so this almost always wins, but it's the trade-off to watch when tightening parameters.
